# 오랜만에 BFS를 푸는 너에게

> BFS 감이 떨어졌다고 느껴질 때, 문제 풀기 전에 2분만 보고 시작하는 복기 노트.

---

## 1) 먼저, 이 문제가 BFS인지 확인한다

아래 신호가 2개 이상 보이면 BFS를 먼저 의심하면 된다.

- "최단 거리/최소 이동 횟수"를 묻고, 간선 가중치가 모두 동일하다.
- "연결된 덩어리 개수"를 센다.
- 격자에서 상하좌우(또는 8방향)로 퍼져 나간다.
- 시작점에서 레벨(거리) 순서로 탐색하는 게 자연스럽다.

---

## 2) 시작할 때 반드시 세팅하는 4가지

1. `queue` : 탐사할 곳
2. `visited` : 탐사한 곳
3. `dist` (최단 거리 문제일 때) : 거리기록
4. `dx, dy` : 방향 벡터

핵심은 한 줄이다.

```txt
큐에 넣는 순간 visited 처리한다.
```

꺼낼 때 방문 처리하면 중복 삽입으로 성능과 정답이 동시에 흔들린다.

---

## 3) 구현 뼈대 (JS)

```js
const queue = [[startX, startY]];
let head = 0;
visited[startX][startY] = true;

while (head < queue.length) {
  const [x, y] = queue[head++];

  for (let d = 0; d < 4; d++) {
    const nx = x + dx[d];
    const ny = y + dy[d];

    if (nx < 0 || nx >= N || ny < 0 || ny >= M) continue;
    if (visited[nx][ny]) continue;

    visited[nx][ny] = true;
    dist[nx][ny] = dist[x][y] + 1;
    queue.push([nx, ny]);
  }
}
```

---

## 4) 오답이 자주 나는 지점

- 테스트케이스마다 `visited/dist/queue` 초기화를 안 한다.
- 범위 체크보다 배열 접근을 먼저 한다.
- `shift()`를 반복해서 시간 초과가 난다.
- 시작점과 도착점이 같은 예외를 빼먹는다.
- 도달 불가능 처리(`-1` 등) 규칙을 마지막에 정리하지 않는다.

---

## 5) 막혔을 때 확인 질문 5개

1. 지금 푸는 문제가 "최단거리"인지 "연결요소 개수"인지 구분했는가?
2. `visited`를 큐에 넣을 때 처리했는가?
3. 경계 체크를 가장 먼저 하고 있는가?
4. 큐 구현이 `head` 인덱스 방식인가?
5. 출력 포맷(도달 불가, 줄바꿈, 다중 테스트케이스)을 마지막에 검증했는가?

---

## 한 줄 요약

> BFS는 결국  
> **큐 + 방문처리 타이밍 + 범위 체크** 세 가지를 정확히 지키는 문제이다.
