# 오랜만에 투포인터를 푸는 너에게

> 투포인터 감이 떨어졌을 때, 문제 풀기 전에 2분만 보고 시작하는 복기 노트.
>  
> 이번 노트의 핵심 인사이트는 `세 용액(BOJ 2473)` 기준으로 정리했다.

---

## 1) 먼저, 이 문제가 투포인터인지 확인한다

아래 신호가 2개 이상 보이면 투포인터를 먼저 의심하면 된다.

- 정렬 후 포인터 이동으로 답에 가까워질 수 있다.
- "합이 K", "합이 0에 가장 가까운 값", "차이가 M 이상 최소" 같은 문장이 나온다.
- 연속 구간 합/길이 최소·최대 문제다. (슬라이딩 윈도우)
- 브루트포스가 `O(n^2)` 또는 `O(n^3)`이라 시간초과가 예상된다.

---

## 2) 시작할 때 반드시 고정하는 기본 개념 4가지

1. **정렬 필요 여부**
2. **포인터의 의미**
3. **이동 규칙**
4. **불변식(invariant)**

핵심은 한 줄이다.

```txt
포인터를 움직일 때마다 "답에 더 가까워지는 방향"이어야 한다.
```

---

## 3) 투포인터 유형 3가지

### 유형 A) 양끝 포인터 (opposite direction)

- `left`는 왼쪽 끝, `right`는 오른쪽 끝에서 시작
- 정렬된 배열에서 합/차이를 보고 한쪽을 줄이거나 늘린다.
- 대표: 두 수의 합, 세 용액(고정점 + 양끝 포인터)

### 유형 B) 같은 방향 포인터 (sliding window)

- `left`, `right`가 둘 다 오른쪽으로만 이동
- 연속 부분합, 최소 길이, 최대 길이 문제
- 대표: 부분합(BOJ 1806)

### 유형 C) 고정점 + 투포인터

- 바깥 루프에서 하나를 고정하고, 나머지 두 개를 투포인터로 탐색
- `O(n^3)`을 `O(n^2)`로 낮출 때 자주 쓴다.
- 대표: 세 용액(BOJ 2473)

---

## 4) 세 용액으로 보는 실전 인사이트

브루트포스 첫 시도:

- 3중 루프 `O(n^3)` → 시간초과

개선:

- 정렬 후 `fix` 하나 고정
- `left = fix+1`, `right = n-1`
- `sum = arr[fix] + arr[left] + arr[right]`
  - `sum > 0` 이면 `right--`
  - `sum <= 0` 이면 `left++`
- 매 순간 `abs(sum)` 최소값 갱신

핵심 전환:

```txt
"전부 다 해보는 탐색" -> "정렬 기반 단조성으로 버리는 탐색"
```

## 5) 구현 전 체크리스트

1. 정렬을 했는가?
2. 포인터 초기 위치가 문제 조건과 맞는가?
3. 이동 규칙이 단조성과 맞는가?
4. 중복/같은 인덱스 선택을 방지했는가?
5. 정답 갱신 조건(`abs`, `min`, 길이 비교 등)을 정확히 썼는가?

---

## 6) 오답/시간초과가 자주 나는 지점

- 정렬 없이 포인터를 움직여 방향성이 사라진다.
- "0에 가까움" 문제에서 `sum` 자체를 비교하고 `abs(sum)` 비교를 놓친다.
- 포인터 경계(`left < right`)를 깨서 같은 원소를 중복 선택한다.
- `O(n^3)` 브루트포스를 그대로 제출한다.
- 슬라이딩 윈도우 문제인데 양끝 포인터처럼 잘못 움직인다.

> **정렬(또는 단조성) + 포인터 의미 고정 + 이동 규칙 일관성** 문제이다.
